# 当四个 AI analyst 全部看多，PM 却选择减仓——一个多 agent 量化系统的真实决策案例

> 本文记录了我用 AgentScope 搭建的多 agent A 股量化交易系统 EvoTraders 在三个月回测中的一个真实决策片段，以及这背后的系统设计思路。

---

## 背景：为什么要用多 agent

传统量化策略是单一模型输出信号，逻辑是固定的。但真实的投资决策从来不是一个维度的——基本面强不代表现在是好的买点，技术面看涨不代表仓位风险可控。

于是我搭了一个四层的多 agent 系统：

- **四个 Analyst Agent**：分别从基本面、技术面、市场情绪、宏观环境四个维度给出信号
- **Risk Agent**：评估整体风险，给 PM 提软约束建议
- **Portfolio Manager Agent**：综合所有信息，输出最终交易决策
- **RiskGuard（规则引擎）**：硬约束兜底，不可商量

框架用的是 AgentScope，核心是 MsgHub——所有 agent 共享同一个对话上下文，可以看到彼此的分析结果，从而产生真正的讨论，而不是各说各话。

## 2024-02-23：一个有意思的分歧

这一天是三个月回测中净值的最高点，账户总资产 ¥521,741，累计收益 +4.35%。

### Analyst 的判断

**fundamentals analyst** 调取了财务数据：

```
600519.SH (贵州茅台):
  Revenue Growth: 18.0%
  Earnings Growth: 19.2%
  ROE: 34.2%
  Net Margin: 52.5%
```

结论：**Bullish，信心 95%**。净利率 52.5% 在 A 股几乎无对手，基本面扎实。

**technical analyst** 调取了技术指标：

```
600519.SH:
  Close: 1607.77
  SMA20: 1549.17  SMA60: 1550.64
  MACD: 11.587 > Signal: 6.093  → BUY
  RSI14: 63.4
  5-day: +0.5%  10-day: +6.9%  20-day: +6.9%
```

结论：**Bullish，信心 85%**。价格在双均线上方，MACD 金叉，动量强劲。

**sentiment analyst** 补充：
> "Moutai 的看涨不只是技术信号，而是市场共识。北向资金持续净流入，茅台是外资持仓最多的 A 股标的，具备情绪锚点效应。RSI 63.4 反映的是机构有序建仓，而非散户追涨。"

四个 analyst 方向一致：茅台看多。

### PM 的决策

```
600519.SH: SHORT 100股  信心 90%

理由：仓位占比 30.9%，违反 A 股 30% 集中度限制。
DCF 显示 -32.3% 的价值缺口，说明即便基本面强劲也已严重高估。
T+1 解锁确认 100 股今日可卖。全部减仓以解决结构性风险，
同时释放资金用于分散化配置。
```

**PM 做了和所有 analyst 相反的决定。**

---

## 为什么会这样

这里有两个层次的逻辑：

**第一层：analyst 和 PM 回答的是不同问题。**

analyst 回答的是：「这家公司值不值得买？」

PM 回答的是：「我现在的仓位结构合不合理？」

即使茅台是好公司，当它已经占到总仓位的 30.9%，再继续持有就是把鸡蛋放在一个篮子里。PM 的工作是管理整个组合的风险，而不是押注某一只票的基本面。

**第二层：好公司不等于好仓位。**

PM 用了 DCF 估值模型来判断价格是否合理。DCF 的逻辑很直接：把这家公司未来几年预计赚到的钱，折算成今天值多少钱。结果是，茅台按当前股价算，你实际付出的钱比它「理论上值的价值」高了 32%。公司确实好，但市场已经提前把未来几年的增长都定价进去了。analyst 看的是公司质地，PM 看的是「现在这个价格贵不贵」——两个问题，答案可以完全不同。

---

## 这个设计想解决什么问题

单 agent 系统里，所有这些维度（基本面、技术面、仓位风险、估值）被打包进一个 prompt，LLM 很难在它们之间做出有层次的权衡，输出结果往往是模糊的「综合判断」。

多 agent 的设计强迫每个维度有独立的分析者，然后在共享上下文里暴露分歧。PM 看到 analyst 全部看多，仍然选择减仓——这个决策的逻辑是完整且可追溯的，不是黑盒里出来的一个结论。

这是系统里最重要的设计原则：**让分歧可见，而不是掩盖它。**

---

## 可解释性：为什么每天都写 reasoning log

每个交易日结束后，系统把所有 agent 的完整推理轨迹写入日志：

```
logs/daily/2024-02-23_reasoning.txt
```

包含：
- 每个 analyst 的工具调用序列（调了哪些 API、拿到了什么数据）
- Conference 讨论阶段各 agent 的发言
- PM 最终决策的 JSON + reasoning 字段

2 月 23 日的日志总共 1062 行，1800 字的 PM reasoning，完整记录了这次分歧是怎么发生的。

这不是为了存档，而是为了**事后归因**。当某一天的决策出了问题，可以精确定位是哪个 analyst 的判断出了偏差，还是 PM 的权衡逻辑有问题，还是数据本身有问题。

## 三个月回测结果

回测区间：2024-01-02 → 2024-03-28，61 个交易日  
标的：600519.SH（贵州茅台）+ 601398.SH（工商银行）  
初始资金：¥500,000

| | Agent 策略 | 买入持有基准 |
|---|---|---|
| 最终资产 | ¥511,611 | ¥502,516 |
| 总收益率 | **+2.32%** | +0.50% |
| 超额收益 | **+1.82%** | — |
| 最大回撤 | 1.15% | — |
| Sharpe | 0.87 | — |

Sharpe 是衡量「每承担一份风险，赚到多少收益」的指标。简单说，高于 1 是优秀，0.87 意味着收益和风险基本匹配，不算亮眼但也没有靠运气。这个数字的意义在于对比：如果收益率一样，但 Sharpe 更高，说明赚这些钱承担的波动更小。

净值峰值出现在 2 月 23 日（+4.35%），之后 PM 通过减仓锁定了部分利润，避免了 3 月份的回落。

## 还差什么

写完这篇感觉挺好，但有几个问题还是得说清楚，不然这个「回测跑赢基准」的结论站不太稳。

**数据有一个作弊的地方。** 财报数据用的是报告期，但现实里财报是有披露延迟的——茅台的年报通常在次年 3 月底才发，但回测里系统在 1 月就拿到了这些数字。这会让基本面 analyst 的判断看起来比实际更准。这个问题有名字叫「前视偏差」，是量化回测里最常见的坑，我目前没有修。

**成交是理想状态。** 现在假设每次下单都能以开盘价全部成交。真实市场里，一笔大单会影响价格本身，成交价格会比预期差，这叫滑点。这部分没有模拟。

**agent 每天失忆。** 每个交易日开始前，短期记忆会清空。这意味着 agent 看不到自己昨天的判断，也没办法意识到「我这周已经连续三天看错了」。框架本身支持长期记忆（ReMeTaskLongTermMemory），我正在接入，准备做个有记忆和没记忆的对比实验，之后会再写一篇。
